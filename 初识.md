# 前端学习笔记

基于 mock-platform-web 和 frontend-asset-platform 两个项目的学习总结。

# 一、编程语言与运行环境

## Q: JavaScript 和 TypeScript 对比，二者都是用来开发前端项目的吗？

**是的，两者都主要用于前端开发**，但 TypeScript 也能用于后端（Node.js）。

### 核心区别

| 对比项 | JavaScript | TypeScript |
|--------|-----------|------------|
| **类型系统** | 动态类型，运行时才检查 | 静态类型，编译时检查 |
| **关系** | 原生语言 | JS 的超集，编译后生成 JS |
| **运行** | 浏览器直接运行 | 需先编译为 JS 才能运行 |
| **IDE 支持** | 一般 | 更好的代码补全、重构、跳转 |
| **错误发现** | 运行时才暴露 | 编写时即可发现 |
| **学习成本** | 低 | 略高，需学类型语法 |
| **适用项目** | 小型/快速原型 | 中大型项目，团队协作 |

### 简单类比

TypeScript = JavaScript + 类型注解。所有合法的 JS 代码都是合法的 TS 代码。

```js
// JavaScript
function add(a, b) {
  return a + b
}
add(1, '2') // 不报错，但结果异常 "12"

// TypeScript
function add(a: number, b: number): number {
  return a + b
}
add(1, '2') // 编译时报错，参数类型不匹配
```

### 结论

两者都是前端开发语言。当前行业趋势是**新项目普遍采用 TypeScript**，大型项目尤其如此。

## Q: JS/TS 和 Node.js (nodes, nodejs, node.js) 是什么关系？

简单来说：**JS/TS 是语言，Node.js 是运行环境**。

### 类比理解

| 角色 | 类比 | 说明 |
|------|------|------|
| **JavaScript** | 人类语言 | 一门编程语言 |
| **TypeScript** | 加了规范的人类语言 | JS 的超集，编译后变成 JS |
| **浏览器** | 课堂 | JS 最早只能在这里"说话" |
| **Node.js** | 另一个课堂 | 让 JS 也能在服务器端"说话" |

### 具体关系

```
TypeScript ──编译──> JavaScript
                        │
              ┌─────────┴─────────┐
              ▼                   ▼
          浏览器运行           Node.js 运行
         (前端/客户端)        (后端/服务端)
```

- **Node.js = Node.js = NodeJS = node**，只是写法不同，都是同一个东西
- Node.js 基于 Chrome V8 引擎，让 JS 可以脱离浏览器运行
- `npm`、`pnpm` 等**包管理器**都是随 Node.js 一起安装的

### 在实际项目中的体现

- **mock-platform-web**（前端）：写的是 JS，开发时用 `node` 运行 webpack-dev-server，最终产物是浏览器运行的静态文件
- **mock-platform**（后端）：写的是 Java，和 Node.js 无关
- **frontend-asset-platform**（前端）：写的是 TS，用 `node` 运行 vite 构建，编译为 JS 后给浏览器运行

### 一句话总结

**JS/TS 是语言，Node.js 是让 JS/TS 能在服务器端运行的环境。前端项目开发时依赖 Node.js 做构建编译，但最终运行在浏览器中。**

# 二、项目技术栈对比

## Q: frontend-asset-platform 和 mock-platform-web 项目开发语言一样吗？

**不一样**，技术栈差异很大：

| 对比项 | mock-platform-web | frontend-asset-platform |
|--------|-------------------|-------------------------|
| **语言** | JavaScript | **TypeScript** |
| **Vue 版本** | Vue **2** (2.6.10) | Vue **3** (3.5.32) |
| **构建工具** | Vue CLI (Webpack) | **Vite** (6.4.2) |
| **UI 框架** | Element UI (2.x) | **Element Plus** (2.13.6) |
| **状态管理** | Vuex | **Pinia** (2.3.1) |
| **路由** | Vue Router 3 | **Vue Router 4** |
| **API 风格** | Options API | Composition API (`<script setup>`) |
| **包管理器** | npm | **pnpm** |
| **Node 要求** | 未指定 | >=20.19.0 |
| **ESLint** | 旧版 .eslintrc | **ESLint 9 flat config** |

总结：**frontend-asset-platform** 是现代 Vue 3 + TypeScript 技术栈，**mock-platform-web** 是老旧的 Vue 2 + JavaScript 技术栈，两者不兼容。

## Q: 两个项目都可以用 Nginx 构建方式吗？

**都可以**，两者编译后都是纯静态文件，都能用 Nginx 托管。

| 项目 | 构建命令 | 产出目录 | 构建工具 |
|------|---------|---------|---------|
| mock-platform-web | `npm run build:prod` | `dist/` | Vue CLI (Webpack) |
| frontend-asset-platform | `pnpm build` | `dist/` | Vite |

唯一的区别在**构建阶段**（需要不同的 Node 版本）：

- **mock-platform-web**：Node 版本要求低，用 npm 安装依赖
- **frontend-asset-platform**：要求 Node >= 20.19.0，用 pnpm 安装依赖，多一步 `vue-tsc` 类型检查

但**运行阶段完全一样**——都是把 `dist/` 目录放进 Nginx 镜像。构建工具的差异只影响开发态，不影响部署态。

```dockerfile
# mock-platform-web 的 Dockerfile
FROM node:16 AS builder
WORKDIR /app
COPY . .
RUN npm install && npm run build:prod

FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/nginx.conf
```

```dockerfile
# frontend-asset-platform 的 Dockerfile
FROM node:20 AS builder
WORKDIR /app
COPY . .
RUN npm install -g pnpm && pnpm install && pnpm build

FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/nginx.conf
```

**构建阶段不同，运行阶段相同**——这就是前端项目部署的通用模式。

前端项目编译后的产物就是**纯静态文件**（HTML、JS、CSS、图片），不需要 Node.js 运行时，任何 Web 服务器（Nginx、Apache、Caddy 等）都能托管。Nginx 是最常用的选择，轻量且高性能。

### 注意事项

- Nginx 配置中需要处理 SPA 路由的 `try_files`（否则刷新页面 404）
- 如需代理后端 API，在 Nginx 中加 `proxy_pass` 即可

## Q: mock-platform-web 可以用 Node 22.18 构建吗？

**大概率可以**，但可能有兼容性警告。

- 项目用的是 `@vue/cli-service@4.4.4`（2020 年发布），当时 Node 版本是 12-14
- Node 22 对旧版包**通常向后兼容**，但不保证 100%

### 可能遇到的问题

| 问题 | 原因 | 解决 |
|------|------|------|
| `npm install` 报错 | 部分旧依赖不兼容 Node 22 | 加 `--legacy-peer-deps` |
| `sass` 编译失败 | node-sass 需要对应 Node 版本二进制 | 改用 `sass`（dart-sass） |
| `ERR_OSSL_EVP_UNSUPPORTED` | Node 17+ 的 OpenSSL 变更 | 设置 `NODE_OPTIONS=--openssl-legacy-provider` |

如果遇到 OpenSSL 报错，构建命令改为：
```bash
NODE_OPTIONS=--openssl-legacy-provider npm run build:prod
```

建议：如果问题太多，可以用 `nvm` 切换到 Node 16 构建，兼容性最稳妥。

# 三、构建工具与包管理

## Q: Vue CLI 和 Vite 分别是什么？

两者都是前端**构建工具**，负责把源码编译成浏览器可运行的静态文件（HTML/JS/CSS）。

### 类比理解

- **Vue CLI (Webpack)** = 老式工厂：流水线长，启动慢，但成熟稳定
- **Vite** = 现代工厂：按需加工，启动极快，是当前主流选择

### 核心区别

| 对比项 | Vue CLI (Webpack) | Vite |
|--------|-------------------|------|
| **诞生时间** | 2018 | 2020 |
| **开发启动** | 慢（需打包整个项目） | 极快（按需编译，毫秒级） |
| **热更新** | 慢（项目越大越慢） | 快（不管项目多大都快） |
| **底层引擎** | Webpack | 原生 ESM + Rollup + esbuild |
| **配置复杂度** | 高（vue.config.js） | 低（vite.config.ts） |
| **维护状态** | **已停止维护** | Vue 官方推荐，活跃开发 |
| **适用 Vue 版本** | Vue 2 | Vue 3（也支持 Vue 2） |

### 为什么 Vite 快？

```
Vue CLI：启动时把所有文件打包 → 再启动服务器 → 你才能看到页面
Vite：   先启动服务器 → 浏览器请求哪个文件 → 才编译哪个文件
```

就像吃自助餐：Vue CLI 是先把所有菜做好再开门；Vite 是先开门，你点什么菜就做什么。

### 在项目中的体现

- **mock-platform-web** 用 Vue CLI → 老项目，Vue 2 时代的标准选择
- **frontend-asset-platform** 用 Vite → 新项目，Vue 3 的标配

**Vue CLI 已被 Vue 官方废弃，新项目应使用 Vite。**

## Q: npm install pnpm@10 -g 解释下这条命令

| 部分 | 含义 |
|------|------|
| **npm** | Node.js 自带的包管理器 |
| **install** | 安装包 |
| **pnpm** | 要安装的包名（另一个包管理器，比 npm 更快更省空间） |
| **@10** | 安装版本号大版本为 10 的最新版（如 10.x.x） |
| **-g** | 全局安装，装完后可在任意目录使用 `pnpm` 命令 |

### 整体意思

**用 npm 全局安装 pnpm 第 10 大版本**，装完后就能在终端直接运行 `pnpm install`、`pnpm build` 等命令。

### 类比

就像用"应用商店"（npm）安装一个"新的应用商店"（pnpm），装完后你可以选择用新商店来安装软件。

### npm vs pnpm 简单对比

- **npm**：每个项目独立下载依赖，重复占磁盘空间
- **pnpm**：依赖只存一份，其他项目通过链接引用，省空间、安装更快

## Q: vue-cli-service build 和 vue-cli-service serve 有什么不同？

两者都是 Vue CLI 提供的命令，用途完全不同：

| 对比项 | `vue-cli-service serve` | `vue-cli-service build` |
|--------|------------------------|------------------------|
| **用途** | 本地开发调试 | 构建生产包部署 |
| **产物** | 无（只在内存中） | `dist/` 目录（静态文件） |
| **是否启动服务器** | 是，启动 dev-server | 否，只输出文件 |
| **热更新** | 有，改代码页面自动刷新 | 无 |
| **代码压缩** | 否（方便调试） | 是（体积更小） |
| **Source Map** | 有 | 无（默认） |
| **运行速度** | 快（按需编译） | 慢（全量构建） |
| **环境文件** | `.env.development` | `.env.production` |

### 简单类比

- **serve** = 试衣间：穿着看效果，随时换，不产生成品
- **build** = 工厂出货：最终生产、打包、发出去

### 在项目中的对应

```bash
npm run dev       → vue-cli-service serve   → 本地开发用
npm run build:prod → vue-cli-service build   → 部署上线用
```

**一句话总结：serve 是给你开发时用的，build 是给你部署时用的。**

## Q: vue-cli-service build --mode staging 这个命令又是干啥？

`--mode` 指定**使用哪个环境文件**来构建：

| 命令 | --mode 值 | 加载的环境文件 | VUE_APP_BASE_API |
|------|-----------|--------------|-----------------|
| `vue-cli-service build` | 默认 production | `.env.production` | `/prod-api` |
| `vue-cli-service build --mode staging` | staging | `.env.staging` | `/stage-api` |

### 为什么需要这个？

同一套代码要部署到不同环境，后端地址不同：

- **生产环境**：前端 → `/prod-api` → 生产后端
- **预发布环境**：前端 → `/stage-api` → 预发布后端

`--mode` 让你用同一份代码构建出不同环境的包。

### 命名规则

`--mode xxx` 会加载 `.env.xxx` 文件，例如：
- `--mode staging` → `.env.staging`
- `--mode test` → `.env.test`
- 不写（默认）→ `.env.production`

### 对应到项目

```bash
npm run build:prod   → build，用 .env.production   → 给线上用
npm run build:stage  → build --mode staging，用 .env.staging → 给测试环境用
```

**一句话总结：`--mode` 就是选择用哪套环境配置来构建。**

## Q: npm run build:prod 和 npm run build 区别是什么呢？

区别在于它们执行的是 `package.json` 中 `scripts` 里定义的**不同命令**，并非 npm 内置功能。

看项目的 package.json：

```json
"scripts": {
  "dev": "vue-cli-service serve",
  "build:prod": "vue-cli-service build",           ← 生产环境构建
  "build:stage": "vue-cli-service build --mode staging"  ← 预发布构建
}
```

### 为什么没有 `npm run build`？

因为这个项目的 scripts 里**没有定义 `build`**，只有 `build:prod` 和 `build:stage`。如果你运行 `npm run build`，会报错 `Missing script: "build"`。

### 对应的环境文件差异

```
.env.production  → VUE_APP_BASE_API = '/prod-api'
.env.staging     → VUE_APP_BASE_API = '/stage-api'
```

不同环境构建出的包，`baseURL` 不同，指向不同的后端地址。

**总结：`npm run xxx` 里的 `xxx` 完全由开发者自定义，叫 `build`、`build:prod`、`build:all` 都行，关键看 scripts 里怎么写的。**

## Q: package.json 的作用是什么？

它是 Node.js 项目的**配置清单**，相当于项目的"身份证"。位于项目根目录。

### 核心作用

| 字段 | 作用 | 本项目示例 |
|------|------|-----------|
| **name** | 项目名称 | `"vue-element-admin"` |
| **version** | 版本号 | `"4.4.0"` |
| **scripts** | 定义快捷命令 | `"build:prod": "vue-cli-service build"` |
| **dependencies** | 运行时依赖 | `vue`, `axios`, `element-ui` 等 |
| **devDependencies** | 开发时依赖 | `@vue/cli-service`, `eslint` 等 |

### 逐项解释

**1. scripts — 定义命令别名**
```bash
npm run build:prod  →  实际执行 vue-cli-service build
npm run dev         →  实际执行 vue-cli-service serve
```

**2. dependencies — 运行必需的包**
项目代码中 `import` 用的库，如 `import Vue from 'vue'`，这些包会被打包进最终产物。

**3. devDependencies — 开发时才需要的包**
如构建工具、测试框架、代码检查工具，只在开发时用，**不会**打包进最终产物。

### 没有 package.json 会怎样？

- `npm install` 不知道装什么依赖
- `npm run dev` 不知道执行什么命令
- 别人拿到代码不知道项目用了哪些库

**一句话总结：package.json 定义了项目"叫什么、怎么跑、依赖什么"。**

# 四、前后端联调与部署配置

## Q: 前端服务 mock-platform-web 如何连接后端 mock-platform？

后端服务信息（从配置文件获取）：
- 端口：`8080`
- Context-Path：`/platform/mock`
- 完整地址：`http://localhost:8080/platform/mock`

当前前端的问题：`request.js` 中 `baseURL` 硬编码为 `https://test-sc-mockplatform.joypaydev.com`，需要改为使用本地代理。

### 需要修改的文件

**1. 修改 `src/utils/request.js`：**
```js
// 将硬编码的远程地址改为相对路径
baseURL: '/platform/mock',
```

**2. 修改 `vue.config.js`，在 `devServer` 中添加 `proxy` 配置：**
```js
devServer: {
  port: port,
  open: true,
  overlay: { warnings: false, errors: true },
  before: require('./mock/mock-server.js'),
  // 添加代理配置
  proxy: {
    '/platform/mock': {
      target: 'http://localhost:8080',
      changeOrigin: true
    }
  }
}
```

不需要 `pathRewrite`，因为后端 context-path 就是 `/platform/mock`，路径原样转发即可。

## Q: 结合 Ingress 配置分析前后端路由

从 Ingress 配置可以看出，域名 `test-mockplatform.jdtest.net` 的路由规则：

| 路径 | 后端服务 | 说明 |
|------|---------|------|
| `/platform/mock` | `mock-platform:80`（后端） | API 请求 |
| `/` | `test-tool-navigator:80`（前端） | 页面请求 |

### 关键发现

线上环境**已经通过 Ingress 实现了前后端路由分离**：
- 前端请求 `/` → 前端 Nginx 容器（静态资源）
- 后端请求 `/platform/mock` → 后端 Spring Boot 容器

### 前端配置建议

既然 Ingress 已经将 `/platform/mock` 路径代理到后端，前端只需将 `baseURL` 指向同域名下的 `/platform/mock` 即可，**无需 Nginx 反向代理**：

```js
baseURL: '/platform/mock',
```

这样：
- **线上**：前端请求 `/platform/mock/xxx` → Ingress 路由到后端 Service
- **本地开发**：前端请求 `/platform/mock/xxx` → webpack-dev-server 代理到 `http://localhost:8080/platform/mock/xxx`

无需修改 Nginx ConfigMap 和 Ingress 配置。

## Q: request.js 中 baseURL 的作用是什么？

`request.js` 中的 `baseURL` 是 axios 的请求前缀，所有 API 请求都会自动拼上它：

```js
baseURL: 'https://test-sc-mockplatform.joypaydev.com'

service.get('/user/list')  →  实际请求 https://test-sc-mockplatform.joypaydev.com/user/list
```

当前配置下，前端**直接请求远程后端地址**，绕过了 Nginx。

### 当前 Nginx 的问题

Nginx 配置**只有静态文件服务，没有 API 代理**：

```
请求路径              Nginx 处理              结果
/                 → 返回静态HTML/JS/CSS    ✅ 页面正常
/platform/mock/xx → 无匹配规则，返回404    ❌ API 不通
```

### 当前架构 vs 推荐架构

**当前（直连远程）：**
```
浏览器 ──页面──→ Nginx（静态文件）✅
浏览器 ──API──→ https://test-sc-mockplatform.joypaydev.com（直连远程）✅ 但绕过了 Nginx
```

**推荐（通过 Ingress 统一）：**
```
浏览器 ──/──────────→ Ingress → Nginx（前端静态文件）✅
```

## Q: .env.production 文件中的 VUE_APP_BASE_API = '/prod-api' 配置是什么？

这个值在构建时会被注入到代码中，作为 axios 请求的**基础路径前缀**。

### 作用链路

```
.env.production 定义                代码使用                       实际请求
VUE_APP_BASE_API='/prod-api'  →  baseURL: process.env.VUE_APP_BASE_API  →  请求 /prod-api/xxx
```

### 具体例子

当 request.js 中设置 `baseURL: process.env.VUE_APP_BASE_API` 时：

```js
// 发起请求
service.get('/user/list')

// 实际发出的请求地址
/prod-api/user/list
```

`/prod-api` 只是前缀，最终需要被代理或 Nginx 转发到真正的后端地址。

### 不同环境的值

| 环境文件 | VUE_APP_BASE_API | 用途 |
|---------|-----------------|------|
| `.env.development` | `/dev-api` | 本地开发 |
| `.env.staging` | `/stage-api` | 预发布 |
| `.env.production` | `/prod-api` | 生产 |

### 但项目有个问题

当前 `request.js` 中 `baseURL` 是硬编码的：
```js
baseURL: 'https://test-sc-mockplatform.joypaydev.com'
```

并没有使用 `process.env.VUE_APP_BASE_API`，所以 `.env.*` 文件中的配置**实际没有生效**。

## Q: 本地开发时 vue.config.js 加代理后的访问链路演示

假设前端页面有一个"获取用户列表"的功能，API 调用代码为：

```js
service.get('/user/list')
```

### 配置改动

**1. `vue.config.js` 加代理：**
```js
devServer: {
  proxy: {
    '/platform/mock': {
      target: 'http://localhost:8080',
      changeOrigin: true
    }
  }
}
```

**2. `request.js` 改 baseURL：**
```js
baseURL: '/platform/mock'
```

### 完整链路

```
① 代码调用
   service.get('/user/list')

② axios 拼接 baseURL
   /platform/mock + /user/list = /platform/mock/user/list

③ 浏览器发起请求
   GET http://localhost:9527/platform/mock/user/list

④ webpack-dev-server 匹配代理规则
   路径以 /platform/mock 开头 → 命中代理规则

⑤ 代理转发到后端
   GET http://localhost:8080/platform/mock/user/list

⑥ 后端 Spring Boot 处理
   context-path=/platform/mock，收到 /user/list 请求

⑦ 后端返回数据 → 代理 → 浏览器 → 页面渲染
```

### 图示

```
┌──────────┐     ┌──────────────────┐     ┌──────────────┐
│  浏览器    │     │ dev-server(:9527) │     │ 后端(:8080)   │
│          │     │   + proxy 代理    │     │ /platform/mock│
└────┬─────┘     └────────┬─────────┘     └──────┬───────┘
     │                    │                       │
     │ GET /platform/mock/user/list               │
     │───────────────────>│                       │
     │                    │ 转发到 localhost:8080  │
     │                    │──────────────────────>│
     │                    │                       │
     │                    │   返回 {code:200,...} │
     │                    │<──────────────────────│
     │   返回响应          │                       │
     │<───────────────────│                       │
     │                    │                       │
```

### 关键点

- 浏览器只跟 `localhost:9527`（dev-server）通信
- dev-server 代理把 `/platform/mock` 开头的请求转发到 `localhost:8080`
- 对浏览器来说，就像请求的是同一个服务器，**不存在跨域问题**

## Q: 后端会收到 /user/list 路径吗？

**会的**，后端收到的完整路径是 `/platform/mock/user/list`。

因为 Spring Boot 配置了 `server.servlet.context-path=/platform/mock`，它会自动处理：

```
后端收到的请求路径：/platform/mock/user/list

Spring Boot 去掉 context-path 后：/user/list

匹配到 Controller 中的 @RequestMapping("/user/list")
```

### 代理没有改写路径

```js
proxy: {
  '/platform/mock': {
    target: 'http://localhost:8080',
    changeOrigin: true
    // 没有 pathRewrite，路径原样转发
  }
}
```

- 浏览器请求：`/platform/mock/user/list`
- 代理转发到：`http://localhost:8080/platform/mock/user/list`
- 路径**原样保留**，没有增删

### 如果用了 pathRewrite 会怎样？

```js
pathRewrite: { '^/platform/mock': '' }
```

这样会把 `/platform/mock` 前缀去掉，后端收到的是 `/user/list`，但 Spring Boot 的 context-path 就匹配不上了，反而会 **404**。

所以当前配置**不需要 pathRewrite**，保持路径原样转发即可。
